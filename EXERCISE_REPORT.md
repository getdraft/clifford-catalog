# Clifford DRAFT v1.0 Readiness Exercise - Final Report

**Exercise window:** 2026-06-21 (single session, fully autonomous)
**Subject framework:** `getdraft/draftsman` - originally v0.60.0 (commit `8a1f3bbb5354d11b8bfaa80d37402a5dafb247ec`); updated 2026-06-22 to the unmerged `fix/inline-relationships` branch, then updated again the same day to v0.61.0 once that branch merged into `main` (commit `2d25d16a7776d6a9e725408cbe3b7704ab5b0819`) - see "Version Update" sections below
**Subject company:** Clifford (fictionalized analogue of Red Rover K12, a Brighton Park Capital-backed K-12 workforce/HCM SaaS company)

---

## Executive Summary

DRAFT's core object model is sound: a five-team, six-repo adoption was completed end-to-end without any human intervention, producing real, cross-referenced YAML across five RequirementGroups, ten shared services, five product lines (each with a product_component, and four with a data_component), and five SoftwareDeploymentPatterns - all committed with a commit history that tells the actual story of the adoption. But the exercise also produced an apparently-complete deliverable that was silently broken: the workspace's own `CODEOWNERS` template calls a team-first catalog layout "recommended", but the GitHub Pages browser generator only reads an "object-type" layout and skips anything else with no warning, so the published catalog browser initially rendered with zero SDPs and zero components despite a clean commit history - caught only when a human opened it, not during authoring. That single defect, more than any of the other real schema gaps found (no tagged releases to lock `framework.lock` against, an undocumented `applicability` dict shape, no way to model scheduled/batch workloads or build-time-only library dependencies), is the reason DRAFT is not ready to call itself v1.0: a framework that can silently ship a broken result while every individual commit looks correct cannot be trusted by an adopter who isn't already deeply familiar with its internals. The single most important fix is closing that gap - either make the generator read both layouts or stop calling the unsupported one "recommended" - with shipping a tagged release for `framework.lock` a close second.

**Post-delivery addendum (same session, after the report above was first published):** three further defects surfaced once a human actually used the published browser and the framework's own validator, rather than reading the commit log. First, all five company-authored RequirementGroups were correctly built with `activation: workspace` but were never added to `.draft/workspace.yaml`'s `requirements.activeRequirementGroups` list - the one mechanism that actually flips a workspace-activation group to "active" in the UI - so the browser displayed five fully-conformant compliance groups as inactive, with no warning anywhere that this second, separate file needed editing. Second, the SDP network diagrams and every object's inbound/outbound relationship panel rendered with zero connecting edges, despite every product_component and data_component already declaring its real dependencies (`runtimeSpec.dependencies`, `runsOn`) with structured refs and purpose text - because the browser's topology and `sdpConnections` rendering is wired exclusively to first-class `relationship` catalog objects, a separate object type nothing prompted us to author even though the dependency data already existed. Third, and most concerning: running `framework/tools/validate.py` for the first time this session - it was never run during the original exercise, and nothing in the authoring workflow or the browser ever surfaces its findings - showed that 26 of the ~40 authored catalog objects fail strict validation outright (missing required top-level fields the schema docs describe as living elsewhere, missing decision-record placeholders, and `mechanism: relationship` requirement dispositions that don't resolve without an undocumented `criteria.capability` wiring). The browser rendered a complete-looking, good-looking catalog throughout; the framework's own compliance engine would have rejected most of it. That disconnect - a pretty UI with zero visibility into the validator that's supposed to gate trust - is now judged the single most damaging finding of the exercise, ahead of the CODEOWNERS/generator mismatch. See Friction Log entries 7-9 and the revised Scorecard below.

**Second post-delivery addendum (same session):** after the activation and relationship-object fixes above were pushed, a follow-up report from a human reviewing the live page - "the SDPs don't effectively express the DAG modeling" - turned out to have a third, more literal cause underneath it. The topology diagram's connecting lines are drawn as an SVG overlay whose CSS class is the connection's raw `protocol` string (e.g. `HTTPS/REST`, `GitHub Actions`, `Aurora PostgreSQL`), but `draft-browser-sdp.css` only assigns a stroke color to the three literal class names `rest`, `amqp`, and `other`, with no default/fallback rule. The result: every one of the 42 relationship objects' connecting lines was rendering into the page with fully correct coordinates and zero color - present in the DOM, invisible on screen. Confirmed directly via a jsdom render of the published page showing nine correctly-positioned `<path>` elements in the People SDP's topology SVG, none of which matched any CSS rule. Fixed by adding a default `stroke`/`background`/`color` rule to the base `.conn-path`/`.proto`/`.pill` classes in `docs/assets/draft-browser-sdp.css` - the framework's own sanctioned local-override layer, which `generate_browser.py` is explicitly written to never overwrite once present. See Friction Log #10.

## Version Update (2026-06-22)

**Action taken:** at Dale's request, re-checked the upstream framework for advancement past the previously-vendored commit before re-evaluating anything. `main` had not moved - still `8a1f3bb`, v0.60.0, identical to what was already vendored. However, `git ls-remote --heads origin` surfaced an open, unmerged branch, `fix/inline-relationships` (tip `839d868`, titled "Resolve DRAFT v1.0 readiness exercise feedback and styling defects"), that is upstream's direct response to this exercise's findings. We adopted it as Clifford's new vendored framework state ahead of merge, with full transparency recorded in `.draft/framework.lock` (`syncedRef: fix/inline-relationships`, `version: "0.59.0-unreleased+inline-relationships"`, `syncedCommit: 839d86898a3e130382596a88879249472fa0ce39`). This is an explicit trade-off: we are now pinned to an unmerged PR, not a tagged release, so Recommended Change #6 (tagged release) remains genuinely open even though almost everything else below closed.

**What the branch actually fixes**, verified by reading the diff and then re-running `validate.py`/`generate_browser.py` against Clifford's real catalog - not just reading the changelog:

- **Friction #6** (CODEOWNERS/team-first layout silently invisible to the browser generator): **Fixed.** `discover_yaml_files()` now does a full recursive scan skipping only tooling/meta folders (`tools`, `schemas`, `docs`, `adrs`, `.github`, `.git`, `.draft`) instead of hardcoding the object-type folder names. Either layout the CODEOWNERS template recommends now actually works.
- **Friction #9 / Recommended Change #2** (`validate.py` never wired into the workflow - previously judged the single highest-priority finding): **Fixed, and empirically confirmed working.** `generate_browser.py`'s `main()` now runs validation first and refuses to generate when it fails, printing `ERROR: DRAFT catalog validation failed. Refusing to generate browser payload. Use --skip-validation to bypass.` We proved this live against Clifford's own catalog (which still fails validation - see below): running it without the flag returned exit code 1 and left `docs/` completely untouched; it only succeeded once we explicitly passed `--skip-validation`. The silent "looks complete, isn't" failure mode that was this exercise's central complaint can no longer happen by accident.
- **Friction #8 / Recommended Change #4** (relationship objects had to be hand-authored a second time even though dependencies were already declared via `runtimeSpec.dependencies`/`runsOn`/`host`): **Fixed, with a new caveat** - see the new Friction #11 below. A new `derive_inline_relationships()` (in a new `tools/uid_utils.py`) now auto-derives virtual `relationship` objects from those existing fields at generate-time, with a deterministic UID so they stay stable across regenerations.
- **Friction #7** (RequirementGroup `activation: workspace` requires a second, undocumented file edit): **Partially fixed.** The schema's `activation` field description and the workspace template's `activeRequirementGroups: []` default now both carry an explanatory comment. The underlying mechanism is unchanged - it's still an opt-in list, not auto-activation - so we backported the equivalent comment into Clifford's own `.draft/workspace.yaml`, since the template-only fix doesn't otherwise reach an already-bootstrapped workspace.
- **Friction #10** (invisible SDP connection lines / CSS): the branch's fix is **byte-identical** to the one we authored and shipped ourselves last session (commit `364324d`) - independent convergence on the same root cause and the same fix.
- **Recommended Change #7** (undocumented `applicability` dict shape): **Fixed.** The schema now documents `applicability` with a worked example (`applicability: { anyOf: [ { field: ..., equals: true } ] }`).
- **Recommended Change #12** (no scaffolded `configurations/requirement-groups/` folder): **Fixed upstream**, matching the manual fix we'd already made to Clifford's own tree in a prior session.
- **Recommended Change #13** (undocumented `workflow` token scope): **Fixed.** `framework/docs/workspaces.md` now carries an explicit `[!IMPORTANT]` callout.
- **Recommended Change #6** (tagged release so `framework.lock` doesn't have to pin to a raw commit): **Still open**, and arguably worse for the moment - we're now pinned to an unmerged branch, which is a step further from a tagged release than `main` was. This is disclosed directly in `framework.lock`'s own comments, not hidden.
- **Recommended Changes #8, #9, #10, #11** (scheduled/batch-task modeling, build-time library dependency type, multi-value auth enum, logical-data-subdivision modeling): **Untouched**, as expected - none were in scope for this branch.

**New finding - Friction #11: auto-derivation and hand-authored relationships silently double-count the same edge.** `derive_inline_relationships()` has no awareness of, and no deduplication against, manually-authored `relationship` objects describing the same source/target pair from the same underlying dependency declaration. Regenerating Clifford's browser against the new framework version produced 59 relationship-type objects (against 42 before) and 18 duplicate source/target pairs in the topology output. For 17 of those, the newly-derived virtual relationship and one of our own hand-authored objects represented the exact same single dependency - e.g. Coverage API's Postgres connection to the Data Platform rendered as two separate lines: our hand-authored "reads/writes via / PostgreSQL wire protocol" edge and a second, auto-derived "coverage logical database" edge, both `FCDCMP9CV4-FHEX -> 5REFR4GN1D-SEAH`. This was confirmed empirically, not just by reading code: we checked which of each duplicate pair's two IDs exists as a YAML file on disk versus only inside the generated `browser-data.js`. The 18th "duplicate" turned out to be a false positive in source/target-only dedup logic - Talent API to People API legitimately has two distinct edges (an async `candidate.hired` event and a synchronous employee-lookup REST call) that happen to share the same source/target pair, and both were correctly kept. Resolution: removed the 17 now-redundant hand-authored `relationship` YAMLs, letting the framework's own auto-derivation be the single source of truth for those edges - which is the entire point of the new feature. Topology edge count after cleanup: 42 (down from 59), with zero duplicate pairs other than the one legitimate Talent/People double-edge. This is a real, reproducible gap in the new feature, logged below as Recommended Change #15, not silently worked around.

**Validation outcome (unchanged by the version bump, as expected):** re-running `validate.py` against Clifford's actual catalog content under the new framework shows the same failure pattern as before - 26 objects failing (missing `failureDomain`/decision-record fields, `mechanism: relationship` resolution failures), plus three RequirementGroups newly failing the just-added `applicability.anyOf`/`allOf` enforcement and one RequirementGroup using an invalid `appliesTo` value. None of this is a framework-version regression - it's Clifford's own pre-existing, already-logged catalog content gaps (Friction #9), now additionally surfaced because validation got *stricter* in this branch (the always-on-group `activeRequirementGroups` check moved from warning to failure, and `applicability` is now actually validated rather than silently accepted as any dict). Fixing Clifford's own catalog content remains future work, separate from the framework defects this exercise was scoped to find.

## Acceptable Use View Investigation (2026-06-22)

**Trigger:** Dale reported the published browser's Acceptable Use Technology view rendering completely blank, and asked both why and what needs to change upstream so the next company onboarding doesn't hit the same thing silently. Traced empirically (live data inspection, not just code reading) rather than guessed.

**Why it's blank:** the view (`acceptableUseGroups()` in `draft-browser.js`) builds its rows by walking every `capability`-type object's `implementations` array and resolving each entry to a TechnologyComponent and lifecycle status. Clifford's published browser does contain capability objects - 44 of them - so they aren't missing. The problem is that all 44 come from the framework's own baseline taxonomy (`framework/configurations/capabilities/*.yaml`), which `validate.py` and `generate_browser.py` unconditionally merge into *every* workspace via a hardcoded `BASE_CONFIGURATION_ROOT` regardless of company. Clifford never authored a single capability object of its own, and every one of those 44 baseline objects ships with `implementations: []` by design - they're generic, vendor-neutral definitions, not company-specific approvals. With zero implementations anywhere, every domain/capability row in the view exists but is empty, for all 44 capabilities, every time.

**How a company is actually supposed to populate it:** via `object_patch` - a workspace-local YAML that targets a baseline capability UID and overlays an `implementations` list (plus ownership) onto it without forking the object (`apply_object_patches()` in both `validate.py` and `generate_browser.py`). The framework even ships four ready-made stub templates for exactly this purpose - `templates/workspace/configurations/object-patches/capability-ownership-{security-identity,data-engineering-quality,compute-runtime,observability}.yaml.tmpl` - pre-filled with the relevant capability UIDs per domain, needing only the company's TechnologyComponent refs and owner contact dropped in.

**The actual defect:** those four templates are dead. `draft_table/repo.py`'s workspace scaffolding creates an empty `configurations/object-patches/` directory (it's listed in `WORKSPACE_DIRS`) but its file-materialization list, `WORKSPACE_TEMPLATE_FILES`, copies `workspace.yaml.tmpl`, `README.md.tmpl`, `AGENTS.md.tmpl`, and seven other root/dotfile templates into a new workspace - and never the four object-patch stubs. They exist only inside the framework's own `templates/` tree, with no command, doc, or onboarding step that surfaces them. Confirmed directly: Clifford's `configurations/object-patches/` contains nothing but a `.gitkeep`. Compounding this, neither `validate.py` nor `generate_browser.py` treats "every merged capability has an empty `implementations` list and zero `object_patch` files exist anywhere in the workspace" as anything other than a perfectly valid, passing state - no failure, no warning. The browser then publishes 44 fully-formed, perfectly-structured domain/capability rows with nothing inside any of them. Not an error. Silence. This is the same failure class as Friction #6 and #9: a catalog that looks complete and validates clean while an entire page of the published site is dead.

**What needs to change upstream, ranked:**
- *Critical:* add the four `capability-ownership-*.yaml.tmpl` stubs to `WORKSPACE_TEMPLATE_FILES` so every new workspace gets them materialized into `configurations/object-patches/` automatically, the same way it already happens for `workspace.yaml.tmpl` and friends. This alone turns "undiscoverable" into "found on day one."
- *Critical:* `validate.py` should warn (or fail) when the merged catalog's capability objects collectively have zero populated `implementations` and zero `object_patch` overlays exist in the workspace - e.g. "N capabilities have no approved TechnologyComponents mapped; the Acceptable Use view will render empty for all of them." Same shape of fix as Recommended Change #2 (wiring validation into the workflow that's actually used).
- *High:* `acceptableUseGroups()`'s consumer should render an explicit empty-state message when every domain group's `rows` is empty, instead of a page with a populated sidebar and no content anywhere - the same gap already fixed for other panels in this framework.
- *Medium:* document `object_patch` and the ownership-stub templates somewhere an adopter would actually encounter them during onboarding - neither `setup-mode.md` nor `capabilities.md` currently mentions either.

Logged as **Friction #12** and **Recommended Change #16** below. Adoption Autonomy's score narrative is updated to reflect this newly-confirmed instance of the same silent-failure pattern that already drives that dimension's score; see Scorecard.

13 of 15 friction points reflect genuine schema/tooling/documentation gaps rather than exercise-specific limitations. Friction #10 is, in one sense, the most concrete finding of the entire exercise: it is direct, reproducible proof that a fully correct data model and a fully correct DOM can still render as "nothing is there" because of a single missing default CSS rule - the same "looks complete, isn't" pattern as Friction #6 and #9, but this time caught by simply looking at the published page rather than reading any tooling output. Friction #9 remains the most severe at the process level: the framework ships a real compliance validator that nothing in the normal workflow ever runs or surfaces, so a catalog can look completely correct in the published browser while failing that validator almost entirely.

## Recommended Changes Before v1.0

1. **[Critical] [RESOLVED in v0.61.0 - see Version Update]** Make `CODEOWNERS.tmpl`'s "team-first" and "object-type" layout options actually equivalent: either teach `generate_browser.py` to read `paths.catalogFolders` by default and document the override, or stop calling the team-first layout "recommended" if the shipped tooling only supports object-type. As built, the template recommends a layout its own browser generator cannot read, with no error when it can't. This is the highest-priority fix because the failure is silent. *(Friction Log: 6 - CODEOWNERS/generate_browser mismatch)*
2. **[Critical] [RESOLVED in v0.61.0 - empirically confirmed working]** Wire `validate.py` into the workflow an adopter actually follows - at minimum, document that it exists and should be run before publishing; ideally, run it automatically before `generate_browser.py` and refuse to publish (or visibly flag, badge-by-badge) any object that fails. Discovered post-delivery that 26 of ~40 authored objects fail strict validation with zero indication from the published browser. This is now judged the single highest-priority fix, ahead of the CODEOWNERS mismatch, because it means the browser's "looks complete" signal and the framework's own compliance signal can disagree entirely with no way to notice. *(Friction Log: 9 - validate.py never surfaced)*
3. **[High] [PARTIALLY RESOLVED in v0.61.0 - doc comment added, mechanism unchanged]** Add a visible note next to `requirement_group.activation: workspace` (in the schema description and in the workspace template's `activeRequirementGroups: []` default) explaining that workspace-activation groups require this second, separate list to be populated before they render as active - or better, default-activate every workspace-scoped group the catalog actually contains rather than requiring an opt-in list with no inline guidance. *(Friction Log: 7 - RequirementGroup activation)*
4. **[High] [RESOLVED in v0.61.0 - see new Friction #11 / Recommended Change #15]** Either auto-generate `relationship` objects from existing `runtimeSpec.dependencies` / `runsOn` refs at build time, or state explicitly (in the schema docs and SKILL.md-equivalent onboarding material) that structured dependency fields elsewhere in the catalog do NOT automatically populate the topology diagram / relationship panels - a second, separate `relationship` object is required per edge even when the dependency is already fully declared elsewhere. *(Friction Log: 8 - SDP topology/DAG edges)*
5. **[Critical] [RESOLVED - independently, both by us in commit `364324d` and byte-identically by `fix/inline-relationships`]** Fix `draft-browser-sdp.css`'s SDP topology connection lines (and the matching connections-table/drawer badges) so they have a default visible color: add a base `stroke`/`background`/`color` rule to `.conn-path`/`.proto`/`.pill` rather than only styling the literal class names `.rest`/`.amqp`/`.other`, since the actual CSS class is the connection's raw protocol string and will not match any of those three for almost any real-world value. As shipped, a fully-correct relationship model renders as a topology diagram with zero visible connecting lines - silently, with no error. *(Friction Log: 10 - invisible SDP connection lines)*
6. **[Critical] [PARTIALLY RESOLVED - `fix/inline-relationships` merged into `main` as v0.61.0, but `main` still has no `v0.61.0` git tag]** Cut a tagged release (e.g. `v1.0.0-rc1`) of the framework repo so adopters can pin `framework.lock`'s `syncedTag` to a real tag instead of a raw commit SHA on `main`. *(Friction Log: 4a - framework.lock/tagging)*
7. **[Critical] [RESOLVED in v0.61.0]** Document the `requirement.applicability` dict's expected shape with at least one worked example in the schema or example catalog, rather than leaving it typed only as an opaque `dict`. *(Friction Log: 4b - applicability)*
8. **[High]** Add a scheduled-task/batch-job-aware construct to the SDP model (e.g. a `cadence` field on deployable-object entries, or a distinct `scheduledTasks` collection alongside `serviceGroups`) so cron/batch workloads aren't forced into the same shape as long-running services. *(Friction Log: 5 - TimePay batch job)*
9. **[High]** Add a build-time/library-dependency relationship or object type so internal shared libraries and infra modules consumed by other teams' SDPs at provisioning time - not deployed as their own running service - are representable in the catalog at all. *(Friction Log: 5 - Platform team)*
10. **[Medium] [RESOLVED in v0.61.0 - applied to Clifford's Identity Gateway, see Version Update]** Allow `authenticationModel` (and similarly-shaped single-value enums) to accept multiple values, or document a supported multi-value pattern, for services that legitimately front more than one auth mechanism simultaneously. *(Friction Log: 4d - Identity Gateway)*
11. **[Medium]** Give `data_store_service` a first-class way to express "one physical store, N logically-owned subdivisions" (e.g. a `logicalDatabases` sub-collection with per-subdivision owner/classification) instead of forcing an all-or-nothing single-object vs. multi-object choice. *(Friction Log: 4d - Data Platform)*
12. **[Medium] [RESOLVED in v0.61.0 - matches our prior manual fix]** Scaffold a `configurations/requirement-groups/` folder in the workspace template, matching the framework's own internal layout convention, so company-authored RequirementGroups have a documented home. *(Friction Log: 4a - configurations scaffold)*
13. **[Low] [RESOLVED in v0.61.0]** State the `workflow` OAuth/PAT scope requirement explicitly in the vendoring/onboarding documentation, since the workflow templates fail silently (a GitHub-side rejection, not a DRAFT-side error) without it. *(Friction Log: 4a - workflow scope)*
14. **[Low]** Reconcile any onboarding/quickstart documentation describing the catalog repo layout with the framework's actual `templates/workspace/` structure, so new adopters are not following two different layouts simultaneously. *(Friction Log: 3 - layout mismatch)*
15. **[High]** Make `derive_inline_relationships()` aware of manually-authored `relationship` objects that already cover the same source/target pair from the same underlying dependency declaration, so the two mechanisms don't silently double-render the same edge as two lines in the topology diagram. 16. **[Critical]** Materialize the four `templates/workspace/configurations/object-patches/capability-ownership-*.yaml.tmpl` stubs into every new workspace's `configurations/object-patches/` directory during scaffolding (add them to `draft_table/repo.py`'s `WORKSPACE_TEMPLATE_FILES`, which currently copies eleven other root/dotfile templates but skips these four entirely), and add a `validate.py` check that warns or fails when the merged catalog's capability objects collectively have empty `implementations` and zero `object_patch` overlays exist anywhere in the workspace. As shipped, a brand-new workspace gets an empty `configurations/object-patches/` folder with no template, no prompt, and no validator signal that anything is missing - so the Acceptable Use Technology view in the published browser renders a fully-formed page of domains and capabilities with nothing inside any of them, for every company, until someone happens to read the framework's own source to discover the stub templates exist. This is the same silent "validates clean, published page is dead" failure class as Friction #6 (CODEOWNERS/layout mismatch) and #9 (validate.py never surfaced), just in a feature area (capability-to-TechnologyComponent mapping) the original exercise never exercised. *(Acceptable Use View Investigation 2026-06-22: Friction #12 - dead object-patch templates)*
