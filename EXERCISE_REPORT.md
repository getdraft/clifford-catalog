# Clifford DRAFT v1.0 Readiness Exercise - Final Report

**Exercise window:** 2026-06-21 (single session, fully autonomous)
**Subject framework:** `getdraft/draftsman` v0.60.0, commit `8a1f3bbb5354d11b8bfaa80d37402a5dafb247ec`
**Subject company:** Clifford (fictionalized analogue of Red Rover K12, a Brighton Park Capital-backed K-12 workforce/HCM SaaS company)

---

## Executive Summary

DRAFT's core object model is sound: a five-team, six-repo adoption was completed end-to-end without any human intervention, producing real, cross-referenced YAML across five RequirementGroups, ten shared services, five product lines (each with a product_component, and four with a data_component), and five SoftwareDeploymentPatterns - all committed with a commit history that tells the actual story of the adoption. But the exercise also produced an apparently-complete deliverable that was silently broken: the workspace's own `CODEOWNERS` template calls a team-first catalog layout "recommended", but the GitHub Pages browser generator only reads an "object-type" layout and skips anything else with no warning, so the published catalog browser initially rendered with zero SDPs and zero components despite a clean commit history - caught only when a human opened it, not during authoring. That single defect, more than any of the other real schema gaps found (no tagged releases to lock `framework.lock` against, an undocumented `applicability` dict shape, no way to model scheduled/batch workloads or build-time-only library dependencies), is the reason DRAFT is not ready to call itself v1.0: a framework that can silently ship a broken result while every individual commit looks correct cannot be trusted by an adopter who isn't already deeply familiar with its internals. The single most important fix is closing that gap - either make the generator read both layouts or stop calling the unsupported one "recommended" - with shipping a tagged release for `framework.lock` a close second.

## Scorecard

| Dimension | Score (1-5) | Narrative |
|---|---|---|
| Schema Completeness | 2 | The core object types (capability, technology_component, the three "*_service" types, product/data components, SDP, requirement_group, decision_record) cover the common case well, but five separate real gaps surfaced in one adoption pass: no multi-value auth-mechanism support, no logical-subdivision concept for a shared data store, no scheduled/batch-task concept distinct from a long-running service, no build-time/library-dependency object type, and an undocumented `applicability` dict shape. A schema this incomplete after a single mid-size adoption suggests more gaps remain undiscovered. |
| Requirement Group Fidelity | 3 | Five inferred control domains (security, data residency, SLA, change management, vulnerability management) mapped onto `requirement_group` cleanly enough to author real, specific requirements with `canBeSatisfiedBy` mechanisms. The model strained at the edges - the undocumented `applicability` field, and an awkward case where a Change Management `cicd-pipeline` requirement had to be argued down to not-applicable for a fully-managed service with no owned pipeline - but did not break. |
| Adoption Autonomy | 1 | A real company would get stuck in at least four places without deep schema-reading or upstream support: locking `framework.lock` with no tagged releases to point to, guessing the `applicability` dict's shape with no example anywhere in the repo, discovering only empirically that workflow templates require a token scope the documentation never calls out, and - most damaging - following the `CODEOWNERS` template's own explicit "recommended" team-first layout straight into a published browser that silently renders without any SDPs or components. That last one is not a case of getting stuck and knowing it; it is shipping a broken deliverable while believing the adoption succeeded, which is a worse failure mode than the others combined. |
| SDP Quality | 3 | All five SDPs are grounded in real repo content (actual stack.yaml dependencies, actual folder structures, actual event-publishing relationships cross-checked across four teams for consistency) rather than templated filler, and two teams raised substantive, specific objections that improved the result. Two of five SDPs (TimePay, Platform) had to be scoped down or marked partially non-compliant because the schema couldn't fully represent what those teams actually do. The content itself held up; what nearly failed was discoverability of that content (see Friction #6 and the Draftsman Discoverability score), not the SDPs' substance. |
| Draftsman Discoverability | 3 | Pointed at five differently-shaped product repos, the inference process produced reasonable, differentiated starting SDPs for all five from `stack.yaml` and folder structure alone. But "discoverability" cuts both ways: the framework's own catalog-browser tool failed to discover 14 of 118 correctly-authored objects because it only scans one of two layouts its own CODEOWNERS template recommends, with no warning when a configured folder is empty or absent. A discoverability score has to account for the framework discovering its own catalog, not just an LLM discovering a product repo, and on that axis it underperformed until manually caught. |

**Overall v1.0 Readiness Score: 2.4 / 5** - Not ready for v1.0. The object model is functional enough to power a real adoption end-to-end, but the density of genuine schema gaps, missing release infrastructure, and - most seriously - self-contradictory template guidance with a silent failure mode, surfaced in a single mid-size company simulation, is too high to credibly ship as a 1.0.

## Recommended Changes Before v1.0

1. **[Critical]** Make `CODEOWNERS.tmpl`'s "team-first" and "object-type" layout options actually equivalent: either teach `generate_browser.py` to read `paths.catalogFolders` by default and document the override, or stop calling the team-first layout "recommended" if the shipped tooling only supports object-type. As built, the template recommends a layout its own browser generator cannot read, with no error when it can't. This is the highest-priority fix because the failure is silent. *(Friction Log: 6 - CODEOWNERS/generate_browser mismatch)*
2. **[Critical]** Cut a tagged release (e.g. `v1.0.0-rc1`) of the framework repo so adopters can pin `framework.lock`'s `syncedTag` to a real tag instead of a raw commit SHA on `main`. *(Friction Log: 4a - framework.lock/tagging)*
3. **[Critical]** Document the `requirement.applicability` dict's expected shape with at least one worked example in the schema or example catalog, rather than leaving it typed only as an opaque `dict`. *(Friction Log: 4b - applicability)*
4. **[High]** Add a scheduled-task/batch-job-aware construct to the SDP model (e.g. a `cadence` field on deployable-object entries, or a distinct `scheduledTasks` collection alongside `serviceGroups`) so cron/batch workloads aren't forced into the same shape as long-running services. *(Friction Log: 5 - TimePay batch job)*
5. **[High]** Add a build-time/library-dependency relationship or object type so internal shared libraries and infra modules consumed by other teams' SDPs at provisioning time - not deployed as their own running service - are representable in the catalog at all. *(Friction Log: 5 - Platform team)*
6. **[Medium]** Allow `authenticationModel` (and similarly-shaped single-value enums) to accept multiple values, or document a supported multi-value pattern, for services that legitimately front more than one auth mechanism simultaneously. *(Friction Log: 4d - Identity Gateway)*
7. **[Medium]** Give `data_store_service` a first-class way to express "one physical store, N logically-owned subdivisions" (e.g. a `logicalDatabases` sub-collection with per-subdivision owner/classification) instead of forcing an all-or-nothing single-object vs. multi-object choice. *(Friction Log: 4d - Data Platform)*
8. **[Medium]** Scaffold a `configurations/requirement-groups/` folder in the workspace template, matching the framework's own internal layout convention, so company-authored RequirementGroups have a documented home. *(Friction Log: 4a - configurations scaffold)*
9. **[Low]** State the `workflow` OAuth/PAT scope requirement explicitly in the vendoring/onboarding documentation, since the workflow templates fail silently (a GitHub-side rejection, not a DRAFT-side error) without it. *(Friction Log: 4a - workflow scope)*
10. **[Low]** Reconcile any onboarding/quickstart documentation describing the catalog repo layout with the framework's actual `templates/workspace/` structure, so new adopters are not following two different layouts simultaneously. *(Friction Log: 3 - layout mismatch)*
