# Clifford DRAFT v1.0 Readiness Exercise - Final Report

**Exercise window:** 2026-06-21 (single session, fully autonomous)
**Subject framework:** `getdraft/draftsman` v0.60.0, commit `8a1f3bbb5354d11b8bfaa80d37402a5dafb247ec`
**Subject company:** Clifford (fictionalized analogue of Red Rover K12, a Brighton Park Capital-backed K-12 workforce/HCM SaaS company)

---

## Executive Summary

DRAFT's core object model is sound: a five-team, six-repo adoption was completed end-to-end without any human intervention, producing real, cross-referenced YAML across five RequirementGroups, ten shared services, five product lines (each with a product_component, and four with a data_component), and five SoftwareDeploymentPatterns - all committed with a commit history that tells the actual story of the adoption. But the exercise also produced an apparently-complete deliverable that was silently broken: the workspace's own `CODEOWNERS` template calls a team-first catalog layout "recommended", but the GitHub Pages browser generator only reads an "object-type" layout and skips anything else with no warning, so the published catalog browser initially rendered with zero SDPs and zero components despite a clean commit history - caught only when a human opened it, not during authoring. That single defect, more than any of the other real schema gaps found (no tagged releases to lock `framework.lock` against, an undocumented `applicability` dict shape, no way to model scheduled/batch workloads or build-time-only library dependencies), is the reason DRAFT is not ready to call itself v1.0: a framework that can silently ship a broken result while every individual commit looks correct cannot be trusted by an adopter who isn't already deeply familiar with its internals. The single most important fix is closing that gap - either make the generator read both layouts or stop calling the unsupported one "recommended" - with shipping a tagged release for `framework.lock` a close second.

**Post-delivery addendum (same session, after the report above was first published):** three further defects surfaced once a human actually used the published browser and the framework's own validator, rather than reading the commit log. First, all five company-authored RequirementGroups were correctly built with `activation: workspace` but were never added to `.draft/workspace.yaml`'s `requirements.activeRequirementGroups` list - the one mechanism that actually flips a workspace-activation group to "active" in the UI - so the browser displayed five fully-conformant compliance groups as inactive, with no warning anywhere that this second, separate file needed editing. Second, the SDP network diagrams and every object's inbound/outbound relationship panel rendered with zero connecting edges, despite every product_component and data_component already declaring its real dependencies (`runtimeSpec.dependencies`, `runsOn`) with structured refs and purpose text - because the browser's topology and `sdpConnections` rendering is wired exclusively to first-class `relationship` catalog objects, a separate object type nothing prompted us to author even though the dependency data already existed. Third, and most concerning: running `framework/tools/validate.py` for the first time this session - it was never run during the original exercise, and nothing in the authoring workflow or the browser ever surfaces its findings - showed that 26 of the ~40 authored catalog objects fail strict validation outright (missing required top-level fields the schema docs describe as living elsewhere, missing decision-record placeholders, and `mechanism: relationship` requirement dispositions that don't resolve without an undocumented `criteria.capability` wiring). The browser rendered a complete-looking, good-looking catalog throughout; the framework's own compliance engine would have rejected most of it. That disconnect - a pretty UI with zero visibility into the validator that's supposed to gate trust - is now judged the single most damaging finding of the exercise, ahead of the CODEOWNERS/generator mismatch. See Friction Log entries 7-9 and the revised Scorecard below.

## Scorecard

| Dimension | Score (1-5) | Narrative |
|---|---|---|
| Schema Completeness | 2 | The core object types (capability, technology_component, the three "*_service" types, product/data components, SDP, requirement_group, decision_record) cover the common case well, but five separate real gaps surfaced in one adoption pass: no multi-value auth-mechanism support, no logical-subdivision concept for a shared data store, no scheduled/batch-task concept distinct from a long-running service, no build-time/library-dependency object type, and an undocumented `applicability` dict shape. A schema this incomplete after a single mid-size adoption suggests more gaps remain undiscovered. |
| Requirement Group Fidelity | 2 | Five inferred control domains (security, data residency, SLA, change management, vulnerability management) mapped onto `requirement_group` cleanly enough to author real, specific requirements with `canBeSatisfiedBy` mechanisms, and the workspace-activation model itself worked correctly once the separate `activeRequirementGroups` list was populated (Friction #7). But `validate.py` shows the authored `requirementImplementation` entries don't actually satisfy the framework's own resolution logic in several places - bare `mechanism: relationship` entries with no `criteria.capability`, and at least one group expecting `mechanism: decisionRecord` where a different mechanism was used - meaning the requirements "looked" satisfied in the browser while failing the actual compliance check (Friction #9). Downgraded from the original 3 once this was discovered. |
| Adoption Autonomy | 1 | A real company would get stuck in at least six places without deep schema-reading or upstream support: locking `framework.lock` with no tagged releases to point to, guessing the `applicability` dict's shape with no example anywhere in the repo, discovering only empirically that workflow templates require a token scope the documentation never calls out, following the `CODEOWNERS` template's own explicit "recommended" team-first layout straight into a published browser that silently renders without any SDPs or components, populating a second undocumented `activeRequirementGroups` list before a correctly-authored RequirementGroup will show as active, and - most damaging of all - authoring an entire catalog with no signal anywhere in the normal workflow that `validate.py` exists or that most of the content fails it. None of these are "stuck and you know it"; every one is "the adoption looks complete and it is not," which is a worse failure mode than a hard blocker. |
| SDP Quality | 2 | All five SDPs are grounded in real repo content (actual stack.yaml dependencies, actual folder structures, actual event-publishing relationships cross-checked across four teams for consistency) rather than templated filler, and two teams raised substantive, specific objections that improved the result. But the diagrams initially rendered with zero connecting edges because dependency data declared in `runtimeSpec.dependencies`/`runsOn` is not automatically reflected as `relationship` objects (Friction #8), and `validate.py` flags structural gaps (missing top-level fields, missing decision-record placeholders) across all five SDPs. Downgraded from the original 3: the content was real, but a meaningfully incomplete fraction of it fails the framework's own validator. |
| Draftsman Discoverability | 3 | Pointed at five differently-shaped product repos, the inference process produced reasonable, differentiated starting SDPs for all five from `stack.yaml` and folder structure alone. But "discoverability" cuts both ways: the framework's own catalog-browser tool failed to discover 14 of 118 correctly-authored objects because it only scans one of two layouts its own CODEOWNERS template recommends, with no warning when a configured folder is empty or absent, and separately rendered zero `activeRequirementGroups`/zero topology edges for reasons invisible until a human went looking (Friction #7, #8). A discoverability score has to account for the framework discovering its own catalog, not just an LLM discovering a product repo, and on that axis it underperformed three separate times in one exercise. |

**Overall v1.0 Readiness Score: 2.0 / 5** - Not ready for v1.0, revised down from an initial 2.4 after two further rounds of post-delivery testing. The object model is functional enough to power a real adoption end-to-end and the content the simulated teams produced was substantively real, not filler - but the gap between "renders correctly in the browser" and "actually satisfies the framework's own compliance validator" turned out to be far wider than the original pass surfaced, and nothing in the standard workflow (the SKILL.md instructions, the vendoring step, or the browser itself) ever calls that gap out. A framework whose visual artifact and whose compliance engine can disagree this completely, with no built-in way to notice, is not ready to be the source of truth a v1.0 label implies.

## Recommended Changes Before v1.0

1. **[Critical]** Make `CODEOWNERS.tmpl`'s "team-first" and "object-type" layout options actually equivalent: either teach `generate_browser.py` to read `paths.catalogFolders` by default and document the override, or stop calling the team-first layout "recommended" if the shipped tooling only supports object-type. As built, the template recommends a layout its own browser generator cannot read, with no error when it can't. This is the highest-priority fix because the failure is silent. *(Friction Log: 6 - CODEOWNERS/generate_browser mismatch)*
2. **[Critical]** Wire `validate.py` into the workflow an adopter actually follows - at minimum, document that it exists and should be run before publishing; ideally, run it automatically before `generate_browser.py` and refuse to publish (or visibly flag, badge-by-badge) any object that fails. Discovered post-delivery that 26 of ~40 authored objects fail strict validation with zero indication from the published browser. This is now judged the single highest-priority fix, ahead of the CODEOWNERS mismatch, because it means the browser's "looks complete" signal and the framework's own compliance signal can disagree entirely with no way to notice. *(Friction Log: 9 - validate.py never surfaced)*
3. **[High]** Add a visible note next to `requirement_group.activation: workspace` (in the schema description and in the workspace template's `activeRequirementGroups: []` default) explaining that workspace-activation groups require this second, separate list to be populated before they render as active - or better, default-activate every workspace-scoped group the catalog actually contains rather than requiring an opt-in list with no inline guidance. *(Friction Log: 7 - RequirementGroup activation)*
4. **[High]** Either auto-generate `relationship` objects from existing `runtimeSpec.dependencies` / `runsOn` refs at build time, or state explicitly (in the schema docs and SKILL.md-equivalent onboarding material) that structured dependency fields elsewhere in the catalog do NOT automatically populate the topology diagram / relationship panels - a second, separate `relationship` object is required per edge even when the dependency is already fully declared elsewhere. *(Friction Log: 8 - SDP topology/DAG edges)*
5. **[Critical]** Cut a tagged release (e.g. `v1.0.0-rc1`) of the framework repo so adopters can pin `framework.lock`'s `syncedTag` to a real tag instead of a raw commit SHA on `main`. *(Friction Log: 4a - framework.lock/tagging)*
6. **[Critical]** Document the `requirement.applicability` dict's expected shape with at least one worked example in the schema or example catalog, rather than leaving it typed only as an opaque `dict`. *(Friction Log: 4b - applicability)*
7. **[High]** Add a scheduled-task/batch-job-aware construct to the SDP model (e.g. a `cadence` field on deployable-object entries, or a distinct `scheduledTasks` collection alongside `serviceGroups`) so cron/batch workloads aren't forced into the same shape as long-running services. *(Friction Log: 5 - TimePay batch job)*
8. **[High]** Add a build-time/library-dependency relationship or object type so internal shared libraries and infra modules consumed by other teams' SDPs at provisioning time - not deployed as their own running service - are representable in the catalog at all. *(Friction Log: 5 - Platform team)*
9. **[Medium]** Allow `authenticationModel` (and similarly-shaped single-value enums) to accept multiple values, or document a supported multi-value pattern, for services that legitimately front more than one auth mechanism simultaneously. *(Friction Log: 4d - Identity Gateway)*
10. **[Medium]** Give `data_store_service` a first-class way to express "one physical store, N logically-owned subdivisions" (e.g. a `logicalDatabases` sub-collection with per-subdivision owner/classification) instead of forcing an all-or-nothing single-object vs. multi-object choice. *(Friction Log: 4d - Data Platform)*
11. **[Medium]** Scaffold a `configurations/requirement-groups/` folder in the workspace template, matching the framework's own internal layout convention, so company-authored RequirementGroups have a documented home. *(Friction Log: 4a - configurations scaffold)*
12. **[Low]** State the `workflow` OAuth/PAT scope requirement explicitly in the vendoring/onboarding documentation, since the workflow templates fail silently (a GitHub-side rejection, not a DRAFT-side error) without it. *(Friction Log: 4a - workflow scope)*
13. **[Low]** Reconcile any onboarding/quickstart documentation describing the catalog repo layout with the framework's actual `templates/workspace/` structure, so new adopters are not following two different layouts simultaneously. *(Friction Log: 3 - layout mismatch)*
